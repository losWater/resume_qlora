# 开发流程 —— 这个项目是怎么一步步搭起来的

按里程碑 M1→M10 推进，每步「先跑通，再优化」。下面记录每步**做了什么、怎么做的、为什么**。

---

## M1：环境搭建

**目标**：让一张 RTX 5060（8GB，Blackwell/50 系新架构）能跑 4-bit 量化训练。

**做了什么**：
- 用 C 盘的纯净母 Python（3.12）建虚拟环境，装在 D 盘项目内：`python -m venv .venv`。
  - 好处：完全隔离，装的包不污染母 Python、不影响其它项目；删项目即删环境。
- **关键坑**：50 系显卡（sm_120）必须装 **CUDA 12.8 版 PyTorch**，普通 `pip install torch` 装的旧版认不出卡。
  - 解决：`pip install torch --index-url https://download.pytorch.org/whl/cu128`
- 其余依赖（transformers/peft/trl/bitsandbytes/accelerate/datasets 等）走 `requirements.txt`。
- 写 `check_env.py` 验收：打印 torch/CUDA 版本、显卡名、并用 `BitsAndBytesConfig(load_in_4bit=True)` 实加载一个小模型，确认 4-bit 链路通。

**对应文件**：`requirements.txt`、`check_env.py`

---

## M2：跑通「文本 → JSON」微调范式（练手）

**目标**：先用一个简单任务（命名实体识别 NER）把整条训练/推理链路跑通，再碰简历。

**为什么先做 NER**：和简历抽取范式完全一样（文本 → JSON），但更简单，用来趟平流程、确认新版库 API 没问题。

**做了什么**：
- `download_model.py`：用 modelscope（国内快）下载基座 **Qwen2.5-1.5B-Instruct**。
- `make_ner_data.py`：本地用模板合成中文 NER 小数据（句子 + 实体 JSON）。
- `train_ner_qlora.py`：QLoRA 微调（下面 M4 详述同套配置），5 分钟训完。
- `infer_ner.py`：加载 adapter 推理，验证模型学会了范式（对没见过的实体也能正确抽取）。

**对应文件**：`download_model.py`、`make_ner_data.py`、`train_ner_qlora.py`、`infer_ner.py`

---

## M3：造简历数据（蒸馏）

**目标**：得到一批高质量的「简历文本 + 标准 JSON」配对数据。简历数据敏感、公开少，所以自己造。

**做了什么**：
- 先定死 **JSON Schema**（`schema.py`）：name / education[] / skills[] / work_years / experience[]，三处共享，保证一致。
- `gen_resume_data.py`：调 **DeepSeek API** 批量生成。
  - **蒸馏思路**：让强模型先编一份虚构简历文本，再输出对应 JSON，成对保存。
  - **多样性**：每条随机注入「行业 × 学历 × 资历」组合（15×4×5），防过拟合。
  - **质量控制**：每条过 `validate_label()` 校验 schema，不合格自动重试。
  - 并发 8 线程，生成 300 条，成功 299。
- `clean_resume_data.py`：清洗噪声（如学历「学士」→「本科」、「金融硕士」→「硕士」），统一到 4 个标准学历，再切 train/val（9:1）。

**为什么用合成数据**：无隐私、合规；可批量、可控字段分布。面试主动说明「学习用途、合成数据」。

**对应文件**：`schema.py`、`gen_resume_data.py`、`clean_resume_data.py`、`.env`（放 API Key）

---

## M4：QLoRA 微调简历抽取（核心）

**目标**：用简历数据微调出能把简历转 JSON 的模型。

**做了什么**（`train_resume_qlora.py`，复用 M2 同套配置，只换数据/指令/输出目录、max_length 调大到 1024）：

- **QLoRA 的「Q」= 4-bit 量化加载基座**（`BitsAndBytesConfig`）：
  - `load_in_4bit=True` + `nf4` 量化类型 + `bf16` 计算 + 双重量化。把 1.5B 模型权重压到约 1/4 显存。
- **QLoRA 的「LoRA」**（`LoraConfig`）：
  - 冻结原模型，只在注意力 q/k/v/o 层挂低秩矩阵学习增量；`r=8, alpha=16`。可训练参数 < 全模型 1%。
- **8GB 显存适配**：
  - `batch=1` + `gradient_accumulation_steps=8`（等效 batch 8，用累积换显存）。
  - `gradient_checkpointing=True`（省显存换时间）。
  - `optim="paged_adamw_8bit"`（优化器状态分页，显存不够时挪到内存）。
  - 验证集 batch 也设 1（默认是 8，eval 时容易爆显存）。
- **completion-only**：`completion_only_loss=True`，只对答案 JSON 算 loss，屏蔽问题部分——让模型专注学「怎么输出 JSON」。
- 训练器用 trl 的 `SFTTrainer`，3 个 epoch，约 5.7 分钟，显存峰值 ~7GB。
- 只保存 LoRA adapter（约 4MB），不存全量权重。

**对应文件**：`train_resume_qlora.py`

---

## M5：评估（微调前 vs 微调后）

**目标**：用数字证明微调有效——这是简历/面试的关键。

**做了什么**（`evaluate.py`）：
- 在验证集上，分别用「原始 Qwen 直接 prompt」和「微调后模型」抽取。
- 技巧：一次加载模型，先评估纯基座，再挂 adapter 评估微调后。
- 指标：JSON 可解析率、姓名/年限精确匹配准确率、技能/教育/工作经历的集合 P/R/F1。
- 输出对比表，存 `outputs/eval_result.json`。
- 结果：列表字段平均 F1 68.1%→79.5%，JSON 可解析率 96.6%→100%。

**对应文件**：`evaluate.py`

---

## M6：岗位匹配模块（规则，不训练）

**目标**：拿简历 JSON 和 JD 算匹配分。

**为什么用规则不用微调**：匹配「分数对不对」缺标注、难客观评估，硬用模型容易翻车；规则可解释、稳定。这种「该用模型用模型、不该用用规则」的判断本身是工程能力。

**做了什么**（`matcher.py`）：
- 简历用微调模型解析；JD 用基座模型（`disable_adapter()` 临时关 LoRA）零样本解析——一个模型双角色，省显存。
- 打分：技能重合率 60% + 年限达标 20% + 学历达标 20% → 0~100 分，输出逐项理由。

**对应文件**：`matcher.py`

## M8：超参数消融实验

**目标**：给超参数选择补上实验依据——之前的配置是社区经验值，现在用控制变量法验证每个选择。

**为什么做**：单次实验只能证明"这套配置有效"，不能回答"为什么是这套配置"。消融实验固定其他变量、每次只改一个维度，指标变化就能明确归因到那个因素上。

**做了什么**（`run_ablation.py`）：
- 以当前配置为中心，6 组对比：r ∈ {4, 8, 16}（alpha 跟随 2r）、挂载层 {QKVO, QKVO+MLP}、lr ∈ {1e-4, 2e-4, 5e-4}。
- 固定种子 42，同一份 train/val，同样 3 epoch；评估函数直接 import `evaluate.py`，与 M5 口径一致。
- **工程设计**：每组训练放独立子进程跑（脚本自调用 `--worker N`），保证组间显存彻底释放；每组结果落盘一个 json，中断后重跑自动跳过已完成的组。
- 每组记录：配置、可训练参数量、训练时长、torch 峰值显存、全套评估指标 → 汇总 `outputs/ablation_results.json`。

**结论**（详见 README 消融表）：
- r 4→8→16 单调提升但边际递减（77.9%→78.6%→80.4%），r=8 未到容量拐点。
- 挂载层加 MLP 收益最大（平均 F1 +4.2pp），且集中在最短板的工作经历字段（+11.6pp）；代价是参数 ×4.2、时长 +23%。
- lr=1e-4 欠收敛（最差且可解析率下滑），2e-4~5e-4 都在稳定区间。
- 单种子 + 29 条验证集，1pp 以内的差异视为噪声（baseline 重训 78.6% vs 最初 79.5% 即此量级）。

**对应文件**：`run_ablation.py`、`outputs/ablation_results.json`

## M9：分布外（OOD）评估

**目标**：把"train/val 同源、分数偏乐观"这个已知局限从一句话变成一个数字。

**为什么做**：同源验证集只能证明模型学会了任务范式，不能回答"分布变了会掉多少"。造一个分布外集是量化这个偏差最廉价的方式（真实简历有隐私成本，仍留在改进计划）。

**做了什么**：
- `gen_ood_data.py`：生成 40 条 OOD 验证集，两条去同源手段同时用——
  1. 换生成模型：deepseek-chat → deepseek-reasoner；
  2. 换模板：每条随机注入 2~3 个"脏"特征（口语化流水账、字段缺失、时间线断档、公司缩写、技能夹在描述里、错别字、时间格式混乱）。
  - 标注仍严格 schema（缺失填空串、错别字只在文本不进标注），过 `validate_label` + 学历规范化，40/40 生成成功，人工抽查 8 条确认标注质量。
- `eval_ood.py`：同一个微调后模型在同源 val 和 OOD 上跑同一套指标（口径与 M5 一致）。

**结论**（详见 README「分布外评估」一节）：
- 平均 F1 79.5%→74.4%（-5.2pp）＝同源乐观偏差的量级；JSON 可解析率两边都 100%，格式服从对分布偏移是稳的，退化在内容抽取。
- 最大崩塌：工作年限准确率 89.7%→47.5%（-42pp），断档/模糊时间线让年限推算失效。
- 反直觉发现：工作经历 F1 反而更高（63.0%→77.7%），因为 OOD 标注照抄文本里的公司写法、归一化难度低——提醒对比要按字段解释，OOD 集构造本身会影响结论。

**对应文件**：`gen_ood_data.py`、`eval_ood.py`、`data/resume_ood.jsonl`、`outputs/ood_eval_result.json`

## M10：DPO 偏好对齐

**目标**：把对齐训练从"了解"变成"实践过"——SFT 只见过正确答案，DPO 用偏好对补上"什么是错的"这个训练信号。

**做了什么**：
- `gen_dpo_data.py`：构造偏好对。chosen=标准答案；rejected=原始基座零样本的真实错误输出（270 条推理：4 全对 / 252 可解析但内容错 / 14 不可解析），优先取"可解析但内容错"的 200 条——这类比完全乱码的学习信号强。用真实错误而不是人工编错误，是因为 DPO 压制的应该是模型真会犯的错。
- `train_dpo.py`：trl `DPOTrainer`，在 SFT adapter 之上继续训练（标准 SFT→DPO 流程），beta=0.1、lr=1e-5、2 epoch。
- **真实踩坑**：第一次跑每步 390 秒（全程要 4~5 小时）。nvidia-smi 定位到显存 7860/8151——DPO 双份（policy/reference）× 双样本（chosen/rejected）的 logits（词表 15 万，一份近 1GB）顶爆 8GB，Windows 驱动静默溢出到共享内存。两招解决：`precompute_ref_log_probs=True`（reference 前向挪到训练前一次算完）+ `max_length` 按实测最长样本（772 token）收紧到 800。速度恢复到 7 秒/步，5.5 分钟训完。另外 `ref_model=None` 让 trl 用 `disable_adapter()` 当 reference，一份权重双角色，本来就是 8GB 卡的前提。
- `eval_dpo.py`：SFT vs SFT+DPO 在同源 val 和 OOD 双集对比。

**结果（负结果，如实记录）**：同源 79.5%→80.4%（+0.9pp，噪声边缘），OOD 74.4%→74.5%（持平），工作经历字段没动。**原因**：rejected 来自基座，但 DPO 施加在 SFT 之上，SFT 已修掉了那些错误模式——偏好对教的是模型已经不犯的错（训练 reward accuracy 轻松 100% 即此信号）。改进方向是用 SFT 模型的 on-policy 错误构造偏好对。

**对应文件**：`gen_dpo_data.py`、`train_dpo.py`、`eval_dpo.py`、`data/resume_dpo.jsonl`、`outputs/dpo_adapter/`、`outputs/dpo_eval_result.json`

---

## 一句话总结技术栈

> Python 3.12 + PyTorch 2.11(cu128) + HuggingFace(transformers/peft/trl/bitsandbytes) + QLoRA 微调 Qwen2.5-1.5B + DeepSeek 蒸馏造数据 + 规则匹配，全部在 8GB 单卡跑通。
